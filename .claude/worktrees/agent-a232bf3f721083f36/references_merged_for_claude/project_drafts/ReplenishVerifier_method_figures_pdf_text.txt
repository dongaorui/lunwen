0.1   Abstract
Large language models (LLMs) have shown promising capability in generating
optimization models and solver code from natural language descriptions. Existing
LLM‑for‑operations‑research methods typically rely on result‑level supervision
signals, such as whether the generated code is executable, whether the solver returns
a feasible or optimal solution, whether the final objective value matches a reference
answer, or whether multiple candidates are mutually consistent. However, these
signals provide limited insight into whether the generated optimization model
contains the correct internal structure. This limitation is particularly critical for
inventory replenishment optimization, where domain‑specific structures such as
inventory balance, order variables, shortage variables, capacity constraints, fixed
ordering costs, binary order‑trigger variables, and Big‑M linking constraints directly
determine model semantics.
In this paper, we propose ReplenishVerifier, an LP‑structure‑grounded process
supervision framework for LLM‑based replenishment optimization modeling.
Given solver code generated by an LLM, ReplenishVerifier executes the code,
exports the induced linear programming (LP) representation, parses the LP
file, and verifies whether the resulting mathematical model satisfies inventory‑
specific structural rules. The framework combines solver execution feedback
with domain‑structured LP verification to support candidate selection, structure‑
aware self‑repair prompt generation, and preference‑data construction for future
lightweight alignment. We further introduce a replenishment‑oriented benchmark
covering five representative problem families, including single‑period newsvendor,
single‑item multi‑period replenishment, shortage‑aware replenishment, multi‑item

                                           1
capacity‑constrained replenishment, and fixed‑order‑cost replenishment with Big‑
M constraints. Experiments are designed to compare Direct generation, Best‑of‑K
selection, Solver‑Filter, Structure‑Only verification, ReplenishVerifier‑Full, and
ReplenishVerifier‑Repair under main, ablation, low‑resource, and difficulty‑wise
settings. Experimental results will be reported after running the full benchmark and
LLM candidate generation pipeline. [TO FILL: main quantitative findings]
Keywords: LLM for optimization modeling; inventory replenishment; LP structure
verification; process supervision; solver‑in‑the‑loop learning; operations research.



0.2   1. Introduction
Optimization modeling is a central task in operations research (OR). Given a decision
problem described in natural language, a modeler must identify decision variables,
formulate objectives, impose constraints, and implement the resulting mathematical
program using a solver interface such as PuLP, Pyomo, Gurobi, or COPT. Recent large
language models (LLMs) have demonstrated strong potential in automating parts
of this workflow by translating textual problem descriptions into solver code. This
creates an opportunity to reduce the modeling burden for classical OR tasks and to
build decision‑intelligence systems that combine natural language interfaces with
mathematical optimization.
Despite this progress, LLM‑based optimization modeling remains fragile. A generated
solver program may be syntactically executable while still representing the wrong
mathematical model. For example, in an inventory replenishment problem, a model
may include an order quantity variable and produce an optimal solver status, yet
omit the inventory balance equation. Similarly, a fixed‑order‑cost replenishment
model may contain a binary variable but fail to link the continuous order quantity
to the binary order‑trigger variable through a Big‑M constraint. In such cases, solver
execution alone may not reveal the semantic error, because the generated model is
still a valid optimization problem, just not the intended one.
Existing LLM‑for‑OR verification strategies mainly provide result‑level signals.
Common signals include: (i) whether the generated code can be executed; (ii)
whether a solver returns a feasible or optimal status; (iii) whether the final objective
value matches a reference value; and (iv) whether multiple generated candidates
agree with one another. These signals are useful but insufficient. They tell us whether
the final result appears correct, but they do not explain which modeling structure is
missing or why a candidate formulation is semantically invalid.
Inventory replenishment optimization provides a natural and practically important
setting for addressing this limitation. Replenishment models exhibit strong domain
structures, including inventory flow conservation, ordering decisions, shortage or
backorder variables, holding and shortage costs, warehouse or ordering capacity
constraints, lead‑time effects, fixed ordering costs, binary order triggers, and Big‑M
linking constraints. These structures are not superficial naming conventions; they
encode the semantics of the decision process. Therefore, a verification framework
that can inspect the internal LP structure of a generated model can provide more
informative supervision than solver status alone.
This paper proposes ReplenishVerifier, an LP‑structure‑grounded process supervision
framework for LLM‑based replenishment optimization modeling. ReplenishVerifier


                                           2
executes LLM‑generated solver code, exports the induced .lp file, parses the objective,
constraints, variables, binary variables, and bounds, and checks inventory‑specific
structural rules. Instead of merely judging whether an answer is correct, the
framework provides structured feedback such as:“the model lacks inventory balance
constraints,”“the problem contains fixed ordering cost but no binary order‑trigger
variable,”or “the model has order variables but no Big‑M linking constraint.”These
feedback signals can be used for candidate selection, repair prompt generation, and
future preference learning.
The main contributions of this work are as follows:
  1. Domain‑structured verification for LLM‑based replenishment modeling. We
     formulate the limitation of result‑only verification in LLM‑for‑OR and introduce
     inventory‑specific LP structure verification as a fine‑grained process supervision
     signal.
  2. ReplenishVerifier framework. We design and implement a modular pipeline
     that executes generated solver code, exports LP files, parses model structure,
     checks replenishment‑specific semantic rules, computes structure‑grounded
     scores, and generates repair feedback.
  3. Replenishment‑oriented benchmark design. We construct a benchmark
     specification covering five replenishment problem families with natural
     language descriptions, reference mathematical structures, PuLP/CBC reference
     implementations, reference LP files, and expected structure labels.
  4. Comprehensive experimental protocol. We design main comparisons, ablation
     studies, low‑resource K analysis, and difficulty‑wise evaluation to isolate the
     value of LP structure verification beyond solver‑only filtering and Best‑of‑K
     selection.

0.3     2. Related Work
0.3.1   2.1 LLMs for Optimization Modeling

Recent studies have explored using LLMs to generate mathematical optimization
formulations or solver code from natural language problem statements. These
methods typically prompt an LLM to identify variables, objectives, and constraints,
then produce executable code in a solver interface. Some approaches further use
chain‑of‑thought prompting, self‑correction, or multiple candidate generation to
improve modeling accuracy. Representative lines of work include LLM‑based OR
modeling, solver‑code generation, and benchmark construction for mathematical
programming tasks.
However, most existing systems focus on general OR modeling across diverse
problem types. While this broad setting is valuable, it often limits the granularity of
verification. A generic verifier can check whether code runs or whether the objective
value is correct, but it usually does not know that a replenishment problem should
contain inventory conservation, or that a fixed‑order‑cost model should include
binary trigger variables and Big‑M linking constraints. ReplenishVerifier differs by
narrowing the scope to inventory replenishment optimization and exploiting its

0.3.2   2.2 Solver‑in‑the‑Loop Verification and Learning

Solver‑in‑the‑loop methods use optimization solvers as external verifiers. Given
generated solver code, the system executes the code and observes whether the solver
returns an optimal, infeasible, unbounded, or error status. Objective values can be
compared against reference solutions when available. These signals can support
rejection sampling, candidate selection, self‑repair, or reinforcement/ preference
learning.
Such solver‑informed feedback is important but incomplete. Solver status is a
property of the generated mathematical program, not necessarily of the intended
problem. An incorrect but well‑posed optimization model may still be solved
optimally. Therefore, solver feedback should be complemented with model‑structure
feedback. ReplenishVerifier follows this principle by combining solver execution with
LP structure parsing and domain‑specific rule checking.

0.3.3   2.3 Process Supervision and Preference Learning

Process supervision aims to supervise intermediate reasoning or modeling steps
rather than only final answers. In optimization modeling, process supervision
can correspond to checking whether variables are correctly introduced, whether
constraints reflect the intended decision process, and whether the objective contains
the necessary cost terms. Preference learning methods such as DPO or RL‑style
optimization can use candidate‑level scores to construct chosen/rejected pairs.
ReplenishVerifier provides a practical source of process‑level supervision. Its LP
structure checker produces interpretable labels for missing or present structures,
and these labels can be converted into scalar scores, repair feedback, or preference
pairs. This makes it suitable for low‑resource adaptation, where a small number
of verified examples may improve small or medium‑sized LLMs without expensive
large‑scale reinforcement learning.

0.3.4   2.4 Inventory Replenishment Optimization

Inventory replenishment is a classical OR domain with rich model structures. Basic
variants include the single‑period newsvendor model and multi‑period replenishment
with inventory balance. More advanced variants introduce shortages or backorders,
holding and shortage costs, warehouse capacity constraints, fixed ordering costs,
binary order decisions, Big‑M linking constraints, lead times, service‑level constraints,
and multi‑item or multi‑warehouse flows.
These structures make replenishment a suitable testbed for structure‑aware LLM
verification. Unlike purely generic optimization problems, replenishment tasks have
clear domain semantics that can be mapped to LP‑level structural patterns. This paper
leverages this property to build a vertical benchmark and verification framework.

0.4.1   3.1 LLM‑based Replenishment Modeling Task

parameters such as demand, costs, capacity, and number of periods/ items, and 𝑠∗
denotes the expected model structure labels. Given 𝑑, an LLM generates a candidate
solver program


                                          𝑐 = ℳ(𝑑),

where 𝑐 is expected to implement an optimization model using a solver interface such
as PuLP.
Executing 𝑐 induces an optimization model 𝑃𝑐 , solver status 𝑧𝑐 , objective value 𝑣𝑐 , and

The goal is to select or improve candidate 𝑐 such that the induced model 𝑃𝑐 is not
only executable and numerically correct, but also structurally faithful to the intended
replenishment model.

0.5     4. Method
0.5.1   4.1 Overview

ReplenishVerifier is designed as an LP‑structure‑grounded process supervision
framework. The pipeline consists of six stages:
  1. Candidate generation. An LLM generates one or more solver‑code candidates for
     each natural language replenishment problem.

                                               7
    2. Solver execution. Each candidate program is executed using PuLP/ CBC. The
       system records execution status, solver status, objective value, and error
       messages.
    3. LP export. If execution succeeds, the induced optimization model is exported to
       a .lp file.
    4. LP structure parsing. The LP parser extracts objective terms, constraint names,
       constraint expressions, variable names, binary variables, and bounds.
    5. Replenishment rule checking. Domain‑specific rules check whether expected
       inventory structures are present.
    6. Structure‑grounded scoring and feedback. The framework computes scores
       for candidate selection and generates natural language feedback for missing
       structures.
This process differs from prompt engineering because the supervision signal is not
only encoded in prompts. Instead, ReplenishVerifier derives structured verification
labels from the actual LP model induced by generated code. Thus, the framework
supervises the modeling process through executable model artifacts.

0.5.2   4.2 Benchmark Construction

The benchmark is generated from replenishment templates. Each instance contains:
{

    "natural_language": "string",

The current prototype supports five problem families:
    1. single_period_newsvendor. Single‑item single‑period model with order
       quantity, leftover inventory, shortage, holding cost, and shortage cost.
    2. single_item_multi_period. Single‑item multi‑period replenishment with

0.5.3 4.3 LP Parser

Given a .lp file exported by PuLP, the parser extracts:
  ‧ optimization sense;
  ‧ objective section;
  ‧ constraint names;
  ‧ constraint expressions;
  ‧ variable names;
  ‧ binary variables;
  ‧ bounds.
The parser is intentionally lightweight and prototype‑oriented. It does not implement
a full LP grammar. Instead, it targets the LP format emitted by PuLP/CBC and uses
robust section‑level parsing and regular expressions. This design is sufficient for the
current controlled benchmark and makes the verifier easy to extend.

0.5.4 4.4 Inventory‑Specific Structure Rules

The structure checker maps parsed LP artifacts to replenishment semantics. Examples
include:
  ‧ order_variable: detected if a variable with prefix Q exists.
  ‧ inventory_variable: detected if a variable with prefix I exists.
  ‧ shortage_variable: detected if a variable with prefix B exists.
  ‧ binary_order_variable: detected if a binary variable with prefix Y exists.
  ‧ inventory_balance: detected if constraints are named like inventory_balance_*
    or expressions contain both inventory and order variables in equality constraints.
  ‧ capacity_constraint: detected if constraint names contain capacity.
  ‧ big_m_constraint: detected if constraint names contain big_m, or expressions
    link Q and Y through inequality constraints.
  ‧ holding_cost: detected if inventory variables appear in the objective.
  ‧ shortage_cost: detected if shortage variables appear in the objective.
  ‧ fixed_order_cost: detected if binary order‑trigger variables appear in the
    objective.

0.5.5   4.5 Structure‑Grounded Scoring

For each candidate, ReplenishVerifier computes multiple score components:
    ‧ executable score;
    ‧ feasibility/optimality score;
    ‧ objective score;
    ‧ structure completeness score;
    ‧ semantic consistency score.
The full method uses a weighted score:


             𝑆𝑓𝑢𝑙𝑙 (𝑐) = 𝛼𝐸(𝑐) + 𝛽𝐹 (𝑐) + 𝛾𝑂(𝑐) + 𝛿𝑆𝐶(𝑐) + 𝜂𝑆𝑒𝑚(𝑐),

where 𝐸(𝑐) indicates executability, 𝐹 (𝑐) indicates solver feasibility/ optimality,
𝑂(𝑐) measures objective closeness, 𝑆𝐶(𝑐) is structure completeness, and 𝑆𝑒𝑚(𝑐)

                                           10
is a semantic consistency score derived from missing structures. In the prototype
implementation, the default weights are:


              𝛼 = 0.20,    𝛽 = 0.20,    𝛾 = 0.20,    𝛿 = 0.30,    𝜂 = 0.10.

These weights can be tuned in future work. The key design choice is that structural
correctness contributes explicitly to candidate selection, rather than being treated as
a post‑hoc diagnostic.

0.5.6   4.6 Baseline and Method Variants

We implement six methods.
Direct. Selects the first candidate for each problem without using execution or
structure verification for ranking. This simulates a single‑shot LLM generation
setting.
Best‑of‑K. Given K candidates, selects the first executable candidate in candidate order.
If no candidate is executable, it selects the first candidate. This baseline uses execution
only as a coarse filter.
Solver‑Filter. Selects candidates using solver‑only signals. Executable candidates
are preferred, Optimal status is preferred, and objective closeness to the reference
objective is used for ranking. LP structure score is not used.
Structure‑Only. Selects candidates using only LP structure completeness. This variant
tests whether domain‑specific structure rules provide useful supervision independent
of final objective values.
ReplenishVerifier‑Full. Uses the full score combining execution, feasibility, objective
accuracy, LP structure completeness, and semantic consistency.
ReplenishVerifier‑Repair. Uses the same selection criterion as the full method but
additionally generates repair prompts when expected structures are missing. The
current implementation does not call a real LLM for repair; instead, it saves prompts
for future use with models such as Qwen or Claude.

0.5.7   4.7 Repair Feedback Generation

When a candidate misses required structures, ReplenishVerifier converts missing
labels into natural language feedback. For example:
  ‧“你的模型缺少库存平衡约束。应加入类似 𝐼[𝑡] = 𝐼[𝑡 − 1] + 𝑄[𝑡] − 𝑑𝑒𝑚𝑎𝑛𝑑[𝑡] 的约
    束。”
  ‧“题目包含固定订货成本，但你的模型没有 binary 订货变量 𝑌 [𝑡]。”
  ‧“你的模型有订货变量 𝑄[𝑡]，但没有 Big‑M 约束 𝑄[𝑡] ≤ 𝑀 ⋅ 𝑌 [𝑡]。”
These messages provide process‑level supervision and can be used for self‑repair, DPO
data construction, or model debugging.

0.7     6. Analysis
0.7.1    6.1 Why Result‑Level Verification Is Insufficient

A key motivation of ReplenishVerifier is that solver execution can validate the
generated mathematical program but not necessarily the intended problem semantics.
Consider a multi‑period replenishment task. A candidate model may introduce order
variables and minimize ordering cost, but omit inventory variables and inventory
balance constraints. Such a model may still be feasible and optimal, but it is not a
replenishment model in the intended sense.
Similarly, a fixed‑order‑cost model may include fixed costs in the objective but fail
to use binary order‑trigger variables. Alternatively, it may introduce binary variables
but omit Big‑M constraints linking order quantities to those binaries. In these cases,
objective values may accidentally match small test instances, but the model structure
remains wrong and will fail under distribution shift.
LP‑structure‑grounded verification addresses this by inspecting the model artifact
itself. It asks not only“did the solver return an answer?”but also“does the generated
LP contain the structures required by this replenishment problem?”

0.7.2    6.2 Solver Feedback and Structure Feedback Are Complementary

Solver feedback and structure feedback capture different failure modes. Solver
feedback is good at detecting syntax errors, runtime errors, infeasibility, and objective
mismatch. Structure feedback is good at detecting missing semantic components.
Therefore, a full scoring function should combine both.
The expected ablation pattern is:


                                           16
  ‧ Solver‑Filter improves executable and objective‑related metrics but may not
    improve structure completeness.
  ‧ Structure‑Only improves structure completeness but may select models with
    poor objective values.
  ‧ ReplenishVerifier‑Full balances both and should perform best on combined
    metrics.

0.7.3    6.3 Repair Feedback as Process Supervision

The repair module converts missing structures into actionable natural language
feedback. This is useful for three reasons.
First, it makes verification interpretable. Instead of returning a scalar score only, the
system explains which modeling structures are missing.
Second, it enables self‑repair. A generated candidate and its feedback can be passed
back to an LLM to produce a revised candidate.
Third, it supports preference‑data construction. Candidates with higher structure‑
grounded scores can be treated as chosen responses, while candidates with missing
structures can be treated as rejected responses.
The current implementation generates repair prompts but does not yet call a real
LLM for second‑round repair. This design keeps the CPU‑only experimental pipeline
reproducible and separates verifier evaluation from LLM inference.

0.7.4    6.4 Error Categories

We plan to analyze errors using the following categories:
  1. Execution errors: syntax errors, missing imports, invalid solver API usage.
  2. Solver errors: infeasible, unbounded, or no valid objective.
  3. Objective errors: executable and structurally plausible models with incorrect
     objective values.
  4. Missing variable errors: missing 𝑄, 𝐼 , 𝐵, or 𝑌 .
  5. Missing constraint errors: missing inventory balance, capacity, or Big‑M
     constraints.
  6. Cost‑term errors: missing holding, shortage, or fixed ordering costs.
  7. Over‑modeling errors: extra structures not required by the problem.

0.8     7. Limitations
This work has several limitations.
First, the current LP parser is a lightweight parser targeting PuLP‑exported LP files. It
is not a complete parser for all LP/MPS dialects. Extending the parser to support more
solver backends and more complex algebraic forms is important future work.
Second, the current structure rules rely partly on naming conventions such as Q,
I, B, and Y. This is acceptable for controlled benchmark generation and candidate


                                           17
evaluation with instructed naming, but real‑world LLM outputs may use different
variable names. Future work should combine name‑based rules with graph‑based or
coefficient‑pattern‑based LP analysis.
Third, the benchmark currently focuses on five replenishment problem families.
More realistic industrial settings may include stochastic demand, service‑level
constraints, lost sales, multi‑echelon networks, supplier constraints, transportation
costs, lead times, and rolling‑horizon policies.
Fourth, ReplenishVerifier‑Repair currently generates repair prompts but does not
perform actual LLM‑based repair in the CPU‑only prototype. Future experiments
should evaluate second‑round repaired candidates generated by LLMs.
Fifth, the full scoring function uses manually chosen weights. Although these weights
are interpretable, future work could learn them from validation data or replace them
with a trained process reward model.
Finally, structure correctness does not guarantee complete mathematical correctness.
A model may contain all expected high‑level structures but still use wrong coefficients,
wrong indices, or incorrect boundary conditions. Fine‑grained coefficient and index
verification remains an important direction.



0.9   8. Conclusion
We presented ReplenishVerifier, an LP‑structure‑grounded process supervision
framework for LLM‑based replenishment optimization modeling. Unlike result‑
only verification methods that rely mainly on executability, solver status, objective
values, or candidate agreement, ReplenishVerifier inspects the LP structure induced
by generated solver code and verifies inventory‑specific modeling semantics. The
framework parses LP files, checks structures such as inventory balance, order
variables, shortage variables, capacity constraints, binary order triggers, fixed
ordering costs, and Big‑M constraints, and produces structure‑grounded scores and
repair feedback.
The proposed framework reframes verification for LLM‑based OR modeling from final‑
answer checking to domain‑structured process supervision. By focusing on inventory
replenishment, it demonstrates how domain knowledge can be operationalized as
verifiable LP‑level rules. The experimental protocol includes main comparisons,
ablations, low‑resource K analysis, and difficulty‑wise evaluation. Full empirical
results will be filled after running LLM candidate generation and benchmark‑scale
experiments. [TO FILL: final result summary]
In future work, we plan to extend ReplenishVerifier to richer replenishment settings,
support more solver backends, improve variable‑name‑agnostic LP analysis, and use
structure‑grounded scores to construct DPO or other preference‑learning datasets for
low‑resource adaptation of optimization‑modeling LLMs.
