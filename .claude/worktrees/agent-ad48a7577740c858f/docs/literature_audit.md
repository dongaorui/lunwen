# Literature-driven audit for ReplenishVerifier

This audit is based on `references_merged_for_claude/extra_paper_notes/`, `references_merged_for_claude/matrices/`, and selected project drafts. The requested `references/` directory is not present in this checkout; the merged reference directory was used instead. No reference files were modified.

## Summary positioning

ReplenishVerifier should be positioned narrowly: **a replenishment-specific LP-structure supervision layer** for LLM-generated optimization models. It is not the first solver-in-the-loop OR modeling system, not the first LP-artifact reward method, not a full hallucination-audit framework, and not a full repair agent. Its defensible increment is that it operationalizes recurring replenishment structures—inventory balance, shortage, capacity, fixed cost, binary order trigger, Big-M links, and cost terms—as verifiable LP-artifact signals for selection, feedback, and future preference data.

## Feedback signal taxonomy

- **Solver execution:** OptiMUS, SIRL, OR-R1-like, StepORLM, OptiRepair, MAMO, ReplenishVerifier.
- **Objective value:** SIRL and OR-R1-like may use solution quality/consensus; ReplenishVerifier uses reference objective only for final evaluation metrics, never formal selection.
- **LP artifact:** SIRL uses generic LP artifacts; ReplenishVerifier parses LP artifacts for replenishment-specific semantics.
- **Process supervision:** StepORLM/GenPRM train or use process reward models; ReplenishVerifier provides verifiable LP-structure process labels.
- **Hallucination audit:** OptArgus is the closest generic audit baseline; ReplenishVerifier is a domain-specific verifier.
- **Repair feedback:** OptiRepair is the closest repair work; ReplenishVerifier-Repair should be described as structure-aware repair prompt generation unless actual second-round LLM repair is run.
- **Preference / RL training:** SIRL, OR-R1, StepORLM, ORLM. ReplenishVerifier currently prepares scores/preferences; it should not claim completed RL/DPO unless implemented and evaluated.

## Experiment implications

1. The main table should include generic strong baselines: `SIRL-like LP-Stats`, `OptArgus-like Audit`, `OR-R1-like Voting`, and `OptiRepair-like Repair-Prompt`, not only Direct/Best-of-K/Solver-Filter.
2. `reference_objective` must be evaluation-only. Candidate selection can use solver status, code validity, LP artifact stats, objective consensus among candidates, and replenishment structures, but not reference objective.
3. The decisive evidence is a case where generic solver/voting/audit baselines select an executable optimal candidate missing inventory balance, capacity, or Big-M, while ReplenishVerifier selects a structurally complete candidate.
4. Synthetic-candidate smoke tests are useful for pipeline validation but should not be the sole main result. Real LLM candidates are required for the primary claims.
