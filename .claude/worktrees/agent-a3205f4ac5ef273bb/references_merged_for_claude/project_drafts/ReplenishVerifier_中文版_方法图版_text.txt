面向库存补货优化的大语言模型
结构验证增强过程监督方法
LP-Structure-Grounded Process Supervision for LLM-based Replenishment Optimization Modeling
中文图版论文初稿
说明：本文档为中文方法图版初稿。正式投稿前仍需补充真实 LLM 候选生成结果、无泄漏候选选择实验、公开数据集验证以及参考文献条目。
摘要
大语言模型在从自然语言描述生成优化模型和求解器代码方面展现出潜力。现有 LLM-for-OR 方法通常依赖结果级监督信号，例如代码是否可执行、求解器是否返回可行或最优状态、最终目标值是否与参考答案一致、多个候选答案是否相互一致。然而，这些信号很难解释生成模型的内部结构是否正确。对于库存补货优化而言，库存平衡、订货变量、缺货变量、容量约束、固定订货成本、二进制订货触发变量以及 Big-M 连接约束等结构直接决定模型语义。
本文提出 ReplenishVerifier，一个面向 LLM 补货优化建模的 LP 结构验证增强过程监督框架。给定 LLM 生成的求解器代码，ReplenishVerifier 执行代码并导出由代码诱导出的 LP 文件，解析变量、约束、目标函数、二进制变量和边界信息，并使用库存补货领域规则检查模型结构是否完整。该框架将 solver execution feedback 与 domain-structured LP verification 结合起来，可用于候选选择、自修复提示生成和后续轻量级偏好学习数据构造。
关键词：大语言模型；优化建模；库存补货；LP 结构验证；过程监督；solver-in-the-loop learning。
1 引言
优化建模是运筹学中的核心任务。面对一个用自然语言描述的决策问题，建模者需要识别决策变量、构造目标函数、写出约束条件，并使用 PuLP、Pyomo、Gurobi 或 COPT 等求解器接口实现数学规划模型。近年来，大语言模型为自动化优化建模提供了新的可能：模型可以将自然语言问题转换为求解器代码，从而降低建模门槛。
但 LLM 生成的优化模型仍然非常脆弱。一个求解器程序即使能够运行，也不代表它表达了正确的数学模型。例如，在多周期补货问题中，模型可能定义了订货量 Q 并返回最优状态，但完全遗漏库存平衡约束；在固定订货成本问题中，模型可能包含二进制变量 Y，却没有加入 Q <= M Y 的 Big-M 连接约束。此时 solver 只能说明“这个错误模型被求解了”，而不能说明“这个模型符合原题语义”。
库存补货优化是研究结构化验证的合适场景。它具有明显且可形式化的领域结构，例如库存流守恒、订货决策、缺货惩罚、持有成本、仓储容量、提前期、固定订货成本和 Big-M 连接关系。本文的核心思想是：不只让求解器告诉模型“对或错”，而是让求解器导出的 LP 文件和补货结构规则告诉模型“错在哪里”。
本文提出 ReplenishVerifier，将 LLM 生成代码执行后得到的 LP 模型作为可验证中间产物，并从中提取目标函数、变量、约束、二进制变量和变量-约束关联关系，再用补货语义规则检查是否缺少关键建模结构。
2 相关工作
2.1 LLM 自动优化建模
已有研究尝试让 LLM 从自然语言生成数学规划模型或求解器代码。典型思路包括提示模型识别变量、目标与约束，或通过多候选生成、自一致性、执行反馈和自修复提高求解器代码质量。代表性方向包括 ORLM、LLMOPT、OptMATH、OptiMUS、MAMO 与 NL4Opt 等。通用方法覆盖面广，但往往只做结果级验证，难以针对库存补货等强结构领域进行细粒度诊断。
2.2 Solver-in-the-loop 验证
Solver-in-the-loop 方法把优化求解器作为外部验证器，利用代码是否可运行、求解状态是否最优、目标值是否合理等信号筛选候选。OR-R1、SIRL、StepORLM 等方向都体现了可验证反馈和过程监督思想。本文与这些方法的区别在于：我们不仅看 solver 的最终状态，还解析 solver 导出的 LP 文件，并检查库存补货特有结构。
2.3 库存补货优化
库存补货优化包含单周期 newsvendor、多周期库存平衡、多品容量约束、缺货或延期交付、固定订货成本、提前期以及多级库存系统等经典模型。这些模型的结构较强，适合作为 LLM 优化建模可靠性研究的垂直场景。
3 问题定义
给定一个自然语言补货问题 d，LLM 生成一个或多个候选求解器程序 c。执行程序 c 后得到诱导优化模型 P_c、求解状态 z_c、目标值 v_c，以及可选的 LP 文件 l_c。参考实现提供参考目标值 v* 和期望结构标签 s*。目标是选择或修复候选，使其不仅可执行、能被求解，而且在结构上符合补货问题语义。
每个问题的结构标签包括：inventory_balance、order_variable、inventory_variable、shortage_variable、capacity_constraint、binary_order_variable、big_m_constraint、lead_time、holding_cost、shortage_cost、fixed_order_cost 等。
结构完整率定义为：在所有期望存在的结构中，被验证器检测到的比例。目标值准确率则只用于最终评测，正式候选选择阶段不应使用参考目标值，以避免测试集答案泄漏。
4 方法：ReplenishVerifier
4.1 总体框架
ReplenishVerifier 的流程如图 1 所示。系统首先接收自然语言补货问题，由 LLM 生成 K 个候选 PuLP 求解器代码。随后执行每个候选代码，并在执行成功时导出 .lp 文件。LP 解析器从文件中提取变量、约束、目标函数和二进制变量；结构规则检查器判断是否存在库存平衡、容量、Big-M、固定成本等关键结构；最后系统根据执行、最优性、结构完整性和语义一致性进行候选选择，并生成自修复反馈。

4.2 Benchmark 构造
本文设计 ReplenishmentOR 风格的补货建模基准，覆盖五类问题：单周期 newsvendor、单品多周期补货、允许缺货的多周期补货、多品容量约束补货、带固定订货成本和 Big-M 的补货问题。每条样本包含自然语言描述、结构化参数、参考 PuLP 代码、参考目标值、LP 文件路径和期望结构标签。
图 2  ReplenishmentOR Benchmark 构造流程
4.3 LP 结构解析与规则检查
LP 解析器面向 PuLP 导出的 LP 文件，提取目标函数区、约束区、变量名、二进制变量和边界信息。结构规则检查器不只依赖 solver status，而是将 LP 内部结构映射到补货语义：Q 表示订货变量，I 表示库存变量，B 表示缺货变量，Y 表示二进制订货触发变量；inventory_balance、capacity、big_m 等约束名称或结构模式用于识别关键约束。
图 3  LP 结构解析与补货语义规则检查
4.4 结构增强候选选择
给定同一问题的 K 个候选，系统比较 Direct、Best-of-K、Solver-Filter、Structure-Only 与 ReplenishVerifier-Full 等方法。正式实验中，候选选择不使用参考目标值；参考目标值只用于最终评测。Full 方法使用执行状态、求解最优性、LP 结构完整性和语义一致性进行排序，避免只选择“能跑但语义错误”的模型。
图 4  候选选择基线与统一评估协议
4.5 自修复与偏好学习
当模型缺少关键结构时，ReplenishVerifier 会把缺失结构转换成自然语言反馈。例如，若题目要求固定订货成本但候选没有二进制触发变量，则系统生成“应加入 Y[t] 并使用 Q[t] <= M * Y[t] 进行连接”的修复提示。进一步地，高分候选与低分候选可以形成 chosen/rejected 偏好对，用于 LoRA-DPO 或过程奖励模型训练。

5.3 评价指标
Executable Rate：候选代码能正常运行的比例。
Optimal Rate：求解器返回 Optimal 的比例。
Objective Accuracy：预测目标值与参考目标值误差是否小于阈值；仅用于最终评测。
Structure Completeness：期望存在的结构中被识别出的比例。
Inventory Balance Accuracy：需要库存平衡的问题中是否检测到库存平衡。
Constraint Coverage：对所有结构维度进行宏平均覆盖率统计。
Average Runtime：执行、导出 LP、解析和验证的平均耗时。
Repair Feedback Count：平均生成的结构修复反馈条数。
6 原型 Smoke Test 结果
当前代码已经完成端到端 smoke test。该实验使用 synthetic demo candidates 验证流程是否可运行，不应作为正式论文主实验结果。它的作用是证明系统能生成数据、构造候选、执行代码、解析 LP、进行结构验证并输出论文表格。正式论文仍需使用 Qwen/Claude 等真实 LLM 生成候选后重跑实验。
从 smoke test 看，Direct 因第一个候选故意设置为语法错误而失败；Best-of-K 能找到可执行候选，但结构和目标不一定正确；Solver-Filter 能提升目标相关指标，但仍可能选择结构不完整模型；Full 方法在该合成场景下同时取得目标和结构满分。该结果说明原型链路可用，但不能替代真实 LLM 实验。
7 讨论与局限
第一，当前 LP 解析器是面向 PuLP 导出格式的轻量级解析器，并非完整 LP/MPS 语法解析器。第二，当前规则部分依赖 Q、I、B、Y 等变量命名规范，真实 LLM 输出若变量名不规范，可能需要更强的变量语义识别或图结构匹配。第三，结构正确不等于完全数学正确，模型仍可能存在系数、索引或边界条件错误。第四，当前 Repair 模块只生成修复提示，尚未完成真实 LLM 的二次修复闭环。第五，正式候选选择必须避免使用 ground truth objective，以免测试集答案泄漏。
8 结论
本文提出 ReplenishVerifier，一个面向库存补货优化的大语言模型 LP 结构验证增强过程监督框架。与只看代码执行、求解状态或目标值的结果级验证不同，ReplenishVerifier 通过执行生成代码、导出 LP 文件、解析模型结构并检查补货语义规则，将验证推进到模型内部结构层面。该框架可以用于候选选择、自修复提示生成和偏好学习数据构造，为低资源 LLM 优化建模提供一种可解释、可复现、可扩展的研究路线。
下一步工作包括：使用 Qwen3-8B 或 Qwen2.5-7B 生成真实候选；在无答案泄漏设置下重跑主实验和消融；扩展公开 OR 子集验证；加入真实 repair 和 LoRA-DPO 训练；进一步降低变量命名依赖并增强结构图解析能力。
