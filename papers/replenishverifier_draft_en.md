# ReplenishVerifier: LP-Structure-Grounded Verification for LLM-Based Replenishment Optimization Modeling

## Abstract

Large language models (LLMs) can generate optimization formulations and solver code from natural language descriptions, but executability, optimal solver status, or agreement among candidate objective values does not necessarily imply that the generated model captures the intended inventory replenishment semantics. This paper studies a narrower but practically important question: how can solver-exported LP artifacts be used to verify the internal replenishment structure of LLM-generated optimization models?

We introduce **ReplenishVerifier**, an LP-structure-grounded verification method for LLM-based replenishment optimization modeling. Given PuLP code generated by an LLM, ReplenishVerifier executes the code, exports the induced `.lp` artifact, parses variables, constraints, objectives, binary declarations, and bounds, and checks whether replenishment-specific structures are present. These structures include inventory balance, shortage/backlog variables, capacity constraints, fixed ordering costs, binary setup/order-trigger variables, and Big-M linking constraints. The resulting structure evidence can be used for candidate selection, structured error feedback, repair-prompt generation, and future preference-data construction.

ReplenishVerifier is not intended as a general LLM-for-OR framework, nor as a full reproduction of SIRL, OptArgus, OptiRepair, OR-R1, or StepORLM. We position it as a plug-in replenishment-specific LP-structure supervision layer. To isolate its contribution, we compare against lightweight baselines: Direct, Best-of-K, Solver-Filter, OR-R1-like Voting, SIRL-like LP-Stats, OptArgus-like Audit, and OptiRepair-like Repair-Prompt. Formal candidate selection never uses `reference_objective`; reference objectives are used only for final evaluation metrics. ## 1. Introduction

Automatic optimization modeling from natural language is an important problem in LLM-for-OR research. A model must identify decision variables, formulate objectives and constraints, and emit executable code for a solver interface. Prior work has shown that LLMs can benefit from prompting, agentic debugging, solver execution, verifiable rewards, process supervision, and preference or reinforcement learning.

However, inventory replenishment modeling often fails in ways that are not captured by syntax checks or solver status alone. A PuLP program may run successfully and return `Optimal`, while still representing the wrong mathematical model. A multi-period replenishment model may omit inventory balance. A fixed-order-cost model may introduce a binary variable but fail to link the order quantity with the binary trigger via a Big-M constraint. A multi-item capacity-constrained model may ignore warehouse capacity. Such errors change the semantics of the replenishment decision but can remain invisible to generic execution, objective-value agreement, or majority-voting signals.

This paper focuses on the vertical domain of inventory replenishment. Replenishment models have recurring structural patterns: order variables, inventory states, shortage or backlog variables, inventory-flow constraints, capacity limits, fixed ordering costs, binary setup triggers, and Big-M links. These structures often leave traces in solver-exported LP files. Instead of judging only final answers, we extract structural evidence from the LP artifact induced by LLM-generated code and use it to verify whether the candidate model contains the required replenishment semantics.

We propose ReplenishVerifier. The system executes candidate PuLP code, exports the induced `.lp` file, parses objective sections, constraints, bounds, binary declarations, and variable names, and checks the detected structures against the expected structures of the problem. The verification output is converted into a ground-truth-free selection score, structured feedback, and repair prompts. Formal candidate selection does not use the reference objective; the reference objective is used only after selection for metrics such as objective accuracy and relative error.

The contributions of this work are:

1. **A replenishment-structure verification perspective.** We operationalize inventory balance, capacity, shortage, fixed ordering cost, binary setup, and Big-M linking as verifiable LP-artifact signals that complement generic solver feedback.
2. **The ReplenishVerifier pipeline.** We implement a prototype that connects PuLP code execution, LP export, LP parsing, structure checking, candidate scoring, and repair feedback.
3. **Literature-driven baseline design.** To avoid attributing generic solver-in-the-loop, LP-artifact, objective-consensus, hallucination-audit, or repair-prompt signals to our method, we include lightweight SIRL-like, OptArgus-like, OptiRepair-like, and OR-R1-like baselines.
4. **A leakage-aware experimental protocol.** We separate formal selection from final evaluation: selection does not use `reference_objective`; synthetic smoke tests are sanity checks only; main claims require real LLM candidates and evaluated repair outputs.

## 2. Related Work

### 2.1 LLMs for Optimization Modeling

Systems such as OptiMUS study LLM-based optimization modeling, solver-code generation, and debugging. They show that solver interaction can improve automated modeling. ReplenishVerifier shares the solver-in-the-loop setting but has a narrower goal: it is not a general modeling agent, but an LP-structure verification layer for inventory replenishment models.

### 2.2 Solver-Informed Learning and LP Artifact Feedback

SIRL-style work uses solver execution, solution quality, LP artifacts, and verifiable rewards for optimization modeling. OR-R1 / SolveOR-R1 style work emphasizes valid-code rewards, executable rewards, test-time RL, and majority voting. ReplenishVerifier does not claim to be the first method to use solvers or LP artifacts. Instead, it separates generic solver/artifact signals from replenishment-specific semantic structures. Accordingly, our experiments include `SIRL-like LP-Stats` for generic LP-artifact statistics and `OR-R1-like Voting` for test-time executable/valid-code/objective-consensus signals.

### 2.3 Process Supervision and PRMs

StepORLM / GenPRM style methods address credit assignment by introducing process supervision or generative process reward models. ReplenishVerifier provides a different type of process evidence: labels derived from solver-exported LP artifacts rather than free-form reasoning traces or a learned PRM. It should therefore be viewed as a verifiable evidence source for future PRM, DPO, or RL-style training, not as a replacement for process-reward modeling.

### 2.4 Hallucination Auditing and Model Repair

OptArgus studies hallucination detection and structural consistency auditing for LLM-based optimization modeling, while OptiRepair studies diagnosis and repair for supply-chain optimization models. These lines are close to our work. We therefore avoid claiming that ReplenishVerifier replaces them. The distinction is experimental: `OptArgus-like Audit` uses only generic objective/variable/constraint/implementation signals; `OptiRepair-like Repair-Prompt` uses only generic execution/audit repair feedback; ReplenishVerifier additionally checks replenishment-specific structures and uses them for ranking and feedback.

### 2.5 OR Benchmarks and Data Construction

ORLM / OR-Instruct / IndustryOR, OptMATH, NL4Opt, and MAMO provide important datasets, benchmarks, and evaluation protocols for optimization modeling. ReplenishVerifier does not propose a general industrial OR benchmark. Its evaluation focuses on a replenishment-oriented, structure-labeled setting. OptMATH-style data-quality protocols and inventory-like subsets of IndustryOR or MAMO are natural future extensions.

### 2.6 Inventory Replenishment and Inventory RL

Inventory RL, MARLIM, MABIM, and multi-product multi-node inventory management studies highlight the complexity of real replenishment settings, including stochastic demand, lead times, multi-echelon networks, multiple agents, and shared resources. These works primarily learn replenishment policies or evaluate simulation environments. ReplenishVerifier instead verifies LLM-generated optimization formulations. They provide domain motivation rather than direct baselines.

## 3. Problem Formulation

Each replenishment problem instance is represented as:

\[
x=(d,\theta,s^*)
\]

where \(d\) is a natural language description, \(\theta\) contains structured parameters such as demand, costs, capacities, periods, and items, and \(s^*\) is a vector of expected replenishment structures. An LLM generates a candidate PuLP program:

\[
c=\mathcal{M}(d,\theta).
\]

Executing the program induces an optimization model \(P_c\), solver status \(z_c\), candidate objective value \(v_c\), and optionally an exported LP artifact \(\ell_c\). ReplenishVerifier aims to select or diagnose candidates using candidate-observable execution and LP-structure signals, without using the reference objective during formal selection.

### 3.1 Structure Labels

The current prototype supports the following structures:

| Structure | Meaning |
|---|---|
| `order_variable` | order quantity variable, e.g., \(Q\) |
| `inventory_variable` | inventory state variable, e.g., \(I\) |
| `shortage_variable` | shortage/backlog variable, e.g., \(B\) |
| `inventory_balance` | inventory flow conservation across periods/items |
| `capacity_constraint` | warehouse/order/resource capacity constraint |
| `binary_order_variable` | binary setup or order-trigger variable, e.g., \(Y\) |
| `big_m_constraint` | Big-M link between order quantity and binary trigger |
| `holding_cost` | holding-cost term in the objective |
| `shortage_cost` | shortage-penalty term in the objective |
| `fixed_order_cost` | fixed ordering-cost term in the objective |
| `lead_time` | lead-time-related arrival structure |

### 3.2 Selection versus Evaluation

Formal selection may use:

- candidate code format validity;
- execution errors, timeouts, and executable status;
- solver status;
- whether a candidate objective exists;
- objective consensus among candidates for the same problem;
- generic LP artifact statistics;
- generic audit issues;
- replenishment-specific expected structures for ReplenishVerifier variants.

Formal selection must not use:

- `reference_objective`;
- distance between candidate and reference objectives;
- choosing the candidate closest to the reference objective.

The reference objective is used only for final evaluation metrics such as objective accuracy and relative error.

## 4. Method

### 4.1 Pipeline Overview

ReplenishVerifier consists of six stages:

1. **Candidate generation.** An LLM generates one or more PuLP code candidates for each replenishment problem.
2. **Code execution.** Each candidate is executed in a controlled working directory, and the system records executability, solver status, objective value, errors, and runtime.
3. **LP export.** If a model is built successfully, it is exported as a `.lp` file.
4. **LP parsing.** The parser extracts objectives, constraints, constraint names, variables, binary variables, and bounds.
5. **Structure checking.** Expected replenishment structures are checked against the parsed LP artifact.
6. **Scoring and feedback.** The system produces selection scores, structure-completeness scores, missing-structure lists, and repair prompts.

### 4.2 Problem-type-specific Required Structure Schema

The implementation centralizes expected structures in `EXPECTED_STRUCTURES_BY_TYPE`. Each problem type defines `required`, `optional`, and `forbidden` structure sets. The main `structure_score` is computed only over required structures; optional structures are reported in certificates but do not affect the score; forbidden structures are currently explicit metadata for future diagnostics. If a benchmark instance contains explicit `expected_structures`, its truthy keys override the default required set for that instance; otherwise the problem-type schema is used as fallback.

This avoids penalizing structures that are not part of a problem type, such as treating a missing Big-M link as an error for a single-period newsvendor instance.

| Problem type | Required structures | Optional examples |
|---|---|---|
| `single_period_newsvendor` | order, inventory, shortage, holding cost, shortage cost | inventory balance |
| `single_item_multi_period` | inventory balance, order, inventory, holding cost | capacity, lead time |
| `single_item_multi_period_shortage` | inventory balance, order, inventory, shortage, holding cost, shortage cost | lead time |
| `multi_item_capacity` | inventory balance, order, inventory, capacity, holding cost | shortage, lead time |
| `fixed_order_cost_big_m` | inventory balance, order, inventory, binary setup, Big-M, holding cost, fixed cost | capacity, lead time |

### 4.3 LP Parser

The parser is a lightweight section-level parser rather than a complete LP grammar. It targets PuLP-exported LP files and extracts:

- optimization sense;
- objective section;
- subject-to constraints;
- bounds;
- binary declarations;
- variable names.

This design is sufficient for the current prototype but remains dependent on the LP format and naming conventions. PuLP constraints without explicit names are exported as `_C1/_C2`; the implementation now preserves these constraints but treats auto-generated names as non-semantic evidence. The prompt therefore requires explicit string names for all PuLP constraints. Future work can replace the parser with a fuller LP/MPS parser.

The current LPStructureGraph provides only incidence-based auxiliary evidence. It is not a complete graph-matching verifier and cannot fully establish algebraic equivalence. Stronger graph matching and coefficient-pattern verification are left for future work.

### 4.4 Replenishment Structure Rules

Structure detection combines layered evidence:

1. **Variable-name hints.** Standard names such as `Q/I/B/Y` and descriptive names such as `order_qty`, `inventory`, `stock`, `backlog`, and `setup`.
2. **Constraint-name hints.** Names such as `inventory_balance`, `flow`, `capacity`, `link`, `big_m`, and `setup`.
3. **Expression-supported evidence.** Constraint expressions must contain the relevant decision variables and relation patterns, e.g., inventory recurrence, Big-M linking, capacity aggregation, or shortage participation.
4. **Index and incidence evidence.** Lightweight index-consistency checks identify repeated or adjacent inventory states, and LPStructureGraph contributes incidence-based auxiliary evidence.
5. **Magnitude evidence.** Big-M rules optionally inspect whether the largest coefficient is plausibly an upper-bound linking coefficient, while leaving bound-aware validation to future work.

A structure name alone is treated only as weak evidence. ReplenishVerifier assigns high structural confidence only when the name signal is supported by expression-level or graph-level evidence. The rule-level certificate records `evidence_strength`, matched names, matched expressions, index-consistency warnings, magnitude checks, and repair hints. These are heuristic high-level structure checks. They do not prove coefficient correctness, indexing correctness, or boundary-condition correctness.

### 4.5 Selection Score

ReplenishVerifier-Full uses the following ground-truth-free score:

\[
S_{full}(c)=0.25E(c)+0.25Z(c)+0.35SC(c)+0.15Sem(c),
\]

where \(E(c)\) indicates executability, \(Z(c)\) indicates `Optimal` solver status, \(SC(c)\) is the average rule-level score over the current problem type's required structures, and \(Sem(c)\) is a semantic-consistency signal derived from required-only missing structures. This raw score does not use `reference_objective`.

The hard selection gate prevents syntactically well-formed but unsolved or infeasible candidates from being selected, while still preserving their structure certificates for diagnosis and repair. By default, only executable + `Optimal` candidates receive a non-zero formal `selection_score`; allowing merely feasible candidates requires an explicit flag.

The implementation also supports `--use_objective_consensus`, which adds a small candidate-objective-consensus signal for ReplenishVerifier-Full. This signal clusters candidate objectives within the same problem and never uses the reference objective. It should be treated as an appendix ablation unless it becomes part of the final method.

### 4.6 Repair Feedback

When required structures are missing, ReplenishVerifier produces structured feedback. For example, it can identify missing inventory-balance equations, missing capacity constraints, missing Big-M links, or missing fixed-ordering-cost terms when those structures are required for the instance. Repair prompts and error-type analysis consume the same required-only `missing` list, so optional or irrelevant absent structures are not reported as repair targets. The current code includes an entry point for second-round LLM repair generation. However, repair should be reported as actual repair only after repaired candidates are generated and re-evaluated; otherwise it is repair-prompt generation.

## 8. Limitations

1. **Dependence on LP format and naming.** The parser targets PuLP-exported LP files and the structure rules still use variable names, constraint names, and objective-variable participation. Descriptive-name fallbacks improve robustness but do not make the parser solver-agnostic.
2. **Structure correctness is not full mathematical correctness.** A model may contain all high-level structures while still using wrong coefficients, wrong indices, or wrong boundary conditions. The current expression-supported checks are not complete algebraic equivalence verification. They identify whether constraints exhibit basic inventory-recurrence, Big-M-linking, or capacity-aggregation patterns, but they cannot fully guarantee coefficient, time-index, or boundary-condition correctness. Stronger coefficient-pattern verification and graph matching are left for future work.
3. **Limited benchmark coverage.** The current benchmark focuses on five replenishment families and does not yet cover stochastic demand, service levels, lost sales, multi-echelon networks, supplier constraints, transportation costs, or rolling-horizon policies.
4. **Hand-designed weights.** The scoring weights are interpretable but manually chosen. Future work could learn them from validation data or train a structure-aware process reward model.
5. **Repair depends on real LLM behavior.** Repair prompts alone do not demonstrate successful repair; repaired candidates must be generated and re-evaluated.
6. **Complementary to general audit and repair systems.** ReplenishVerifier does not replace OptArgus-like audit, OptiRepair-like repair, solver validation, or objective evaluation. It provides additional replenishment-structure evidence.

## 9. Conclusion

We presented ReplenishVerifier, an LP-structure-grounded verification method for LLM-based replenishment optimization modeling. It exports LP artifacts from generated PuLP code, parses variables, constraints, objectives, binary declarations, and bounds, and checks for replenishment structures such as inventory balance, shortage/backlog, capacity, fixed ordering cost, binary setup/order trigger, and Big-M links. The resulting evidence supports candidate selection, error feedback, repair prompts, and future preference-data construction.

The method is positioned as a replenishment-specific LP-structure supervision layer rather than a general LLM-for-OR framework or a replacement for existing systems. The experimental design uses OR-R1-like Voting, SIRL-like LP-Stats, OptArgus-like Audit, and OptiRepair-like Repair-Prompt to distinguish generic solver/artifact/audit/voting signals from replenishment-specific structure supervision.
